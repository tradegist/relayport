# Exporting a new Pydantic model to the type packages

The type packages (`@tradegist/relayport-types` for TS, `relayport-types` for Python) are generated from Pydantic models declared in `services/`. To expose a new model:

## The one required step

Add the **exported name** (as a string) to `SCHEMA_MODELS` in `schema_gen.py`, under the correct key. `SCHEMA_MODELS` is typed `dict[str, list[str]]`; `schema_gen.py` resolves each name via `getattr(module, name)` at generation time ŌĆö so paste the symbol's name in quotes, not the symbol itself.

```python
SCHEMA_MODELS: dict[str, list[str]] = {
    "shared": [
        "Fill",
        "Trade",
        "OptionContract",
        # "NewPrimitiveModel",  # ŌåÉ add here for CommonFill primitives
    ],
    "relay_core.relay_models": [
        "WebhookPayloadTrades",
        "RunPollResponse",
        "HealthResponse",
        # "NewWebhookPayload",  # ŌåÉ add here for relay API / notifier contracts
    ],
    "market_data.models.dividends": [
        "DividendsUpcomingItem",
        "DividendsUpcomingResponse",
        # "NewMarketDataModel",  # ŌåÉ add here for market data API
    ],
}
```

Then run:

```bash
make types
```

This regenerates:
- `types/typescript/<ns>/types.d.ts` (via `schema_gen.py`)
- `types/typescript/<ns>/index.d.ts` (via `gen_ts_barrels.py`)
- `types/python/relayport_types/<file>.py` (via `gen_python_types.py`)
- `types/python/relayport_types/__init__.py` (auto-AST-walked barrel)

## Choosing the right key

| Key | Maps to TypeScript namespace | Use for |
| --- | --- | --- |
| `"shared"` | `BrokerRelay` | CommonFill primitives (`Fill`, `Trade`, enums, `OptionContract`). Source: `services/shared/models.py` |
| `"relay_core.relay_models"` | `RelayApi` | Notifier payload contracts + relay API responses. Source: `services/relay_core/relay_models.py` (which re-exports from `relay_core/notifier/models.py`) |
| `"market_data.models.dividends"` | `MarketDataApi` | Market data endpoint types. Source: `services/market_data/models/dividends.py` |

If your model conceptually belongs to one of these layers but lives in a different file, **move the model to the right file** instead of registering a new key. Adding new top-level keys requires updating `gen_python_types.py` and `gen_ts_barrels.py` and is rarely correct.

## Supported symbols

Each string name in `SCHEMA_MODELS` must resolve via `getattr(module, name)` to one of these (Pydantic's `TypeAdapter` handles all three):

- **`BaseModel` subclasses** ŌĆö generated as `interface Foo` in TS, `class Foo(BaseModel)` in Python.
- **Discriminated-union `TypeAlias` values** ŌĆö `WebhookPayload = WebhookPayloadTrades | WebhookPayloadOrders` style. The TypeAlias must be defined at module top level so `getattr` finds it.
- **Class aliases** ŌĆö `WebhookPayload = WebhookPayloadTrades` produces `export type WebhookPayload = WebhookPayloadTrades` in TypeScript via an `allOf: [$ref]` schema entry.

In all three cases you only paste the **name** (e.g. `"WebhookPayload"`) into `SCHEMA_MODELS` ŌĆö the symbol itself stays in its source module.

## Pydantic config that propagates to the generated types

- **`ConfigDict(extra="forbid")`** ŌåÆ `additionalProperties: false` in JSON Schema ŌåÆ strict TS types (no `[k: string]: unknown`). **Always use this on external-contract models** (webhook payloads, API responses).
- **No defaults on always-populated fields** ŌĆö a default makes the field optional in JSON Schema ŌåÆ optional in TS (`fillCount?: number`). Defaults only for fields that are legitimately absent.
- **`Field(default_factory=list)`** for mutable defaults, only when the field is genuinely optional. Never bare `[]` or `{}`.

## Things that are NOT required

- ŌØī No need to edit `types/typescript/<ns>/index.d.ts` ŌĆö auto-generated by `gen_ts_barrels.py`.
- ŌØī No need to edit `types/python/relayport_types/__init__.py` ŌĆö auto-generated by AST walk.
- ŌØī No need to edit `types/typescript/index.d.ts` unless you're adding a brand-new namespace (rare).

## Things that ARE manual

- Ō£ģ The top-level barrel `types/typescript/index.d.ts` is **hand-maintained**. It declares the three namespaces (`BrokerRelay`, `RelayApi`, `MarketDataApi`). Add a new namespace here only if you genuinely need a new top-level grouping.
- Ō£ģ Adding a new key to `SCHEMA_MODELS` requires also updating `gen_python_types.py` and `gen_ts_barrels.py` to know about the new source file. **This is rare.**

## Verification

After `make types`:

1. `git diff types/typescript/` should show the new type in `types.d.ts` and re-exported from `index.d.ts`.
2. `git diff types/python/relayport_types/` should show the new class in the appropriate file and re-exported from `__init__.py`.
3. `make typecheck` (which runs `tsc --noEmit` on `types/typescript/`) should pass.

If `make typecheck` fails, the most common cause is a removed/renamed export that the hand-maintained `types/typescript/index.d.ts` still references. Fix the barrel.
